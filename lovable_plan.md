# План реализации «Студия сайтов» (/lovable) — полный проект

Документ описывает целевую архитектуру, границы системы, API, безопасность, превью, агентов и поэтапный ввод. Предполагается развитие текущего стека: **React (Vite)**, **Express (ollama-api)**, **nginx**, **Ollama/внешние LLM**, **Linux VPS**.

---

## 1. Цели и границы

### 1.1 Цель продукта

Пользователь в веб-интерфейсе (аналог Lovable):

- формулирует задачу на создание или доработку сайта/веб-приложения в **контексте проекта** (чат + файлы);
- **согласует с агентом план работ** до начала реализации (см. **1.6**);
- **ожидает** завершения реализации и сборки после согласования;
- видит **живое превью** готовой **рабочей сборки** и может протестировать результат, затем снова обратиться к агенту за правками (**1.6**);
- управляет **очередью задач** и версиями (workspace);
- опционально получает **публичную ссылку** на демо или деплой.

Агент (LLM + оркестратор) получает **ограниченный, аудируемый** набор инструментов: работа с файлами проекта, запуск команд в изолированной среде, проверки (lint/build), выдача URL превью. **Порядок работ обязателен:** сначала план и согласование с пользователем, затем реализация и сборка (**1.6**).

### 1.2 Вне скоупа первых версий (явно)

- Полноценный «бесконечный» хостинг продакшен-сайтов пользователей без лимитов (только контролируемые демо/staging).
- Произвольный произвольный outbound из превью без ограничений (см. безопасность).
- Гарантия 100% генерации без ошибок — нужны fallback, логи и ручной режим.

### 1.3 Принципы

1. **Изоляция по умолчанию**: каждый проект/сессия сборки — отдельная cgroup/контейнер/VM tier по политике.
2. **Минимальные привилегии**: агент не имеет доступа к хосту, только к API runner и workspace API.
3. **Воспроизводимость**: lockfile, зафиксированные образы Docker, версии Node.
4. **Аудит**: кто что запросил, какие команды выполнены, какие файлы изменены.

### 1.4 Ценность для пользователя и планка «премиум»

Разработка считается успешной только если результат **реально полезен** конечному пользователю продукта (а не только «собралось без ошибок»).

- **Ясность и доверие**: понятные формулировки, предсказуемое поведение, осмысленные состояния загрузки/ошибок, отсутствие «мёртвых» экранов.
- **Визуальный и UX-уровень**: согласованная типографика, отступы и сетка, нормальные focus-состояния, анимации без отвлечения от задачи — ориентир на ощущение **качественного B2C/B2B** продукта, а не прототипа «на скорую руку».
- **Адаптивность (обязательно)**: проекты по умолчанию проектируются **mobile-first**; проверка ключевых брейкпоинтов (например 360 / 768 / 1024 / 1280+), тач-таргеты, читаемость без горизонтального скролла там, где это не оправдано.
- **Доступность**: базовая a11y (семантика, контраст, подписи к интерактиву, клавиатура) — не опциональное «если останется время», а часть **Definition of Done** для публичных шаблонов и демо.
- **Производительность**: разумный размер бандла, ленивая загрузка тяжёлых маршрутов где уместно, не блокировать первый экран лишними запросами — см. бюджеты в разделе про инструменты агента.

### 1.5 Технологический фундамент (UI и данные)

**База фронта (зафиксировать как стандарт шаблонов студии):**

- **React 18+** и **Vite** — уже в проде; новые проекты из студии выводить в том же стеке.
- **TypeScript** — по умолчанию для генерируемых приложений (строже типы → меньше регрессий).
- **Tailwind CSS** — основа стилизации; при необходимости **headless/ui**-подход (например **Radix UI**, **React Aria**) для доступных примитивов — по политике allowlist зависимостей.
- **Маршрутизация**: **React Router** (уже в стеке) или согласованный аналог в шаблоне.
- **Серверное состояние**: **TanStack Query** — для запросов, кеша, повторов и синхронизации с API (популярный, хорошо документированный выбор).

**Данные и бэкенд шаблонов (выбор по типу продукта — утвердить на ревью):**

| Вариант | Когда уместно | Плюсы |
|--------|----------------|--------|
| **Supabase** (Postgres + Auth + Storage + Realtime) | Быстрый старт, таблицы, RLS, файлы, «бэкенд в облаке» | Меньше кастомного API; проверенная экосистема |
| **PostgreSQL** (самостоятельно или управляемый) | Полный контроль, сложные схемы, миграции | Стандарт индустрии; совпадает с «идеей» Supabase |
| **MySQL / MariaDB** | Наследие, хостинг-провайдер, команда заточена под SQL | Допустимо как опция шаблона, не как единственный стандарт |
| **SQLite** (в т.ч. через **Drizzle** / **Prisma** локально) | MVP, edge, тесты | Не для высоконагруженной multi-tenant студии без осознанной архитектуры |

Рекомендация для **Studio metadata** (проекты, ревизии, задачи агента): **PostgreSQL** или **Supabase** как управляемый Postgres — одна согласованная модель с возможностью RLS позже. JSON-файлы в `ollama-api/data` — переходный слой до миграции схемы из раздела 3.

**Валидация и контракты API:** **Zod** (или аналог) на границе клиент–сервер; **OpenAPI** при выделении отдельного studio-api.

### 1.6 Основной сценарий: план → согласование → сборка → готовый результат

Это **обязательный** пользовательский поток для типовой задачи агента (новый экран, фича, заметная доработка). Цель — чтобы пользователь не получал «сюрприз-сборку» без понимания объёма, и в конце всегда имел **проверяемый артефакт**.

| Фаза | Что происходит | Кто инициирует |
|------|------------------|----------------|
| **1. План** | Агент формулирует **структурированный план**: цели, экраны/маршруты, данные/API, риски, порядок шагов, критерии готовности. Без записи кода в workspace (кроме уточняющих вопросов в чате). | Агент |
| **2. Согласование** | Пользователь **явно подтверждает** план в UI (кнопка «Согласовать и запустить» / «Внести правки в план» с повторным шагом 1). До подтверждения **запрещено** начинать массовые правки файлов и полноценный build-пайплайн реализации. | Пользователь |
| **3. Реализация и сборка** | После согласования оркестратор переводит задачу в режим **реализации**: инструменты файлов, `run_command`, lint/test/build по **§6.5**, при необходимости **preview**. Пользователь **ожидает** завершения (прогресс в чате / SSE: этапы, логи сборки). | Система + агент |
| **4. Результат** | Выдаётся **готовая рабочая сборка**: успешный `build`, ссылка на **preview** (или эквивалент), краткий **отчёт** (что сделано, что проверено). Пользователь **тестирует** продукт в превью. | Агент → пользователь |
| **5. Итерации** | При необходимости пользователь описывает правки или новую задачу — цикл повторяется **с шага 1** (короткий план для мелкого фикса допустим по политике, но лучше явно зафиксировать даже одношаговый план). | Пользователь → агент |

**Инварианты:**

- **Сборка «всерьёз»** (массовая генерация кода + production build) стартует **только после** согласования плана пользователем.
- Во время фаз 3–4 UI показывает понятное **ожидание** (очередь, спиннер этапов, возможность отмены run — по возможности в следующих версиях).
- Итоговое состояние задачи: **«готово к проверке»** с артефактом сборки, пригодным для ручного теста и последующих правок через того же агента.

## 2. Высокоуровневая архитектура

```
[Браузер]
    │ HTTPS (ollama.site-al.ru)
    ▼
[nginx]
    ├── /           → статика SPA (ollama-web-react)
    ├── /api/chats… → существующий ollama-api (cookie auth)
    └── /api/studio/*  → расширение gateway или отдельный studio-api (:3020)
              │
              ├── PostgreSQL / Supabase (метаданные проектов, ревизии, задачи — целевая схема)
              ├── Object storage или FS (snapshots workspace)
              └── [Runner orchestrator]
                        │
                        ├── очередь задач (BullMQ/Redis или встроенная pg-queue)
                        └── Worker-пул
                                  └── Docker (или Firecracker/Kata при росте)
                                        └── контейнер: Node + vite/pnpm + git worktree
```

**Рекомендация по разбиению сервисов (эволюция):**

| Этап | Состав |
|------|--------|
| MVP | Расширение **ollama-api** маршрутами `/api/studio/*`, один **runner** как child-процесс Docker CLI на том же хосте |
| Рост | Вынос **studio-api** + **runner-worker** на отдельные systemd units / второй машине |
| Scale | Kubernetes Nomad/Temporal для долгих задач |

## 3. Модель данных

### 3.1 Сущности

- **User** — уже есть (`user_id` cookie / сессия gateway).
- **StudioProject** — `id`, `user_id`, `name`, `slug`, `created_at`, `updated_at`, `status` (`draft`, `active`, `archived`).
- **StudioRevision** — снимок файловой системы или git commit hash внутри workspace; `project_id`, `parent_revision_id`, `message`, `created_at`.
- **StudioFile** — опционально нормализованное хранение blob (или только tarball/zstd архив ревизии).
- **StudioTask** — постановка для агента: `id`, `project_id`, `revision_id`, `title`, `payload` (JSON: в т.ч. текст плана и версия после правок), `status` — рекомендуемая машина состояний: `planning` → `awaiting_user_approval` → `implementing` → `building` → `ready_for_review` → `done` \| `failed`; опционально `agent_run_id`.
- **AgentRun** — один запуск LLM с инструментами: `id`, `task_id`, `model`, `prompt_tokens`, `tool_calls` (JSONL), `finished_at`, `error`.
- **PreviewSession** — `id`, `project_id`, `revision_id`, `url`, `expires_at`, `runner_container_id`.

Хранилище файлов:

- **Вариант A (проще)**: каждая ревизия — **tar.gz** в `/var/lib/studio/projects/{project_id}/revs/{revision_id}.tar.gz`.
- **Вариант B**: Git bare внутри сервера + `git archive` для ревизий (удобнее для diff и merge).

### 3.2 Индексы и целостность

- Уникальный `(user_id, slug)` для проекта.
- FK каскады при удалении проекта → задачи, превью, файлы.

### 3.3 Хранилище Studio и пользовательские приложения

- **Метаданные студии** (проекты, ревизии, задачи, прогоны агента) — целевое хранилище: **PostgreSQL** (или Supabase); миграции через **Drizzle** / **Prisma** / **node-pg-migrate** — выбрать один инструмент на репозиторий.
- **Пользовательские приложения**, генерируемые агентом, могут подключать **Supabase client**, **Prisma + Postgres**, или облегчённый **REST + TanStack Query** — в зависимости от выбранного шаблона при создании проекта.
- **Секреты** пользовательских приложений (ключи Supabase и т.д.) — только через зашифрованное хранилище или переменные окружения runner'а, не коммитить в ревизии (см. 7.4).

## 4. API (контракты)

Все под `/api/studio`, авторизация та же что у чатов (cookie `user_id` + проверка владельца).

### 4.1 Проекты

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/studio/projects` | список проектов пользователя |
| POST | `/api/studio/projects` | создать `{ name, slug?, template?: 'vite-react-ts' }` |
| GET | `/api/studio/projects/:id` | метаданные + последняя ревизия |
| PATCH | `/api/studio/projects/:id` | переименование, архив |
| DELETE | `/api/studio/projects/:id` | удалить (с подтверждением на клиенте) |

### 4.2 Файлы и ревизии

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/studio/projects/:id/files` | дерево или flat-list текущей HEAD ревизии |
| GET | `/api/studio/projects/:id/files/**path` | содержимое файла |
| PUT | `/api/studio/projects/:id/files/**path` | записать файл (с optimistic locking `rev`/`etag`) |
| GET | `/api/studio/projects/:id/revisions` | список снимков (meta: id, parent, message, размер архива) |
| POST | `/api/studio/projects/:id/revisions` | зафиксировать снимок `{ parent_revision_id?, message }` |
| PATCH | `/api/studio/projects/:id/workspace` | правки пачкой: `{ base_revision_id?, operations: [...] }` (после approve плана) |

Для больших правок агента предпочтительнее **один PATCH**:

`PATCH /api/studio/projects/:id/workspace`

```json
{
  "base_revision_id": "uuid",
  "operations": [
    { "op": "write", "path": "src/App.tsx", "content_base64": "..." },
    { "op": "delete", "path": "tmp/old.tsx" },
    { "op": "mkdir", "path": "src/components" }
  ]
}
```

Ответ: `{ revision_id, conflicts?: [...] }`.

### 4.3 Задачи и запуск агента

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/studio/projects/:id/tasks` | список задач |
| POST | `/api/studio/projects/:id/tasks` | создать задачу `{ title, prompt, images?: [...] }` |
| POST | `/api/studio/projects/:id/tasks/:taskId/approve-plan` | пользователь **согласовал план** → задача переходит к реализации (**§1.6**, **§6.3**) |
| POST | `/api/studio/projects/:id/agent/run` | запуск агента: для режима «планирование» — только черновик плана; полная реализация — **после** approve-plan (или эквивалент в payload) |

*Оркестратор обязан отклонять опасные операции до статуса согласования (массовые запись файлов, долгие build без апрува).*

Стрим статуса (для UI как чат):

- **SSE**: `GET /api/studio/projects/:id/agent/stream/:runId`
  - события: `delta`, `plan_ready` *(план готов к согласованию)*, `tool_start`, `tool_end`, `revision`, `preview_url`, `error`, `done`.

Альтернатива: WebSocket для двусторонности (остановка run).

### 4.4 Превью

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/api/studio/projects/:id/preview` | `{ revision_id?, mode: 'static' \| 'dev' }` |
| GET | `/api/studio/projects/:id/preview` | текущая активная сессия + URL |
| DELETE | `/api/studio/projects/:id/preview` | остановить контейнер |

Ответ содержит **внутренний или публичный URL**:

- Публичный: `https://preview.{main-domain}/{token}/` через nginx → upstream контейнера.
- Или одноразовый JWT в path для TTL и отзыва.

## 5. Runner и превью (тонкости)

### 5.1 Режимы превью

1. **`static`** — `npm ci && npm run build`, раздача `dist/` через nginx внутри контейнера или через общий nginx с volume.
   - Плюсы: безопаснее, предсказуемый CSP.
   - Минусы: медленнее на каждое изменение.

2. **`dev`** — `npm ci && npm run dev -- --host 0.0.0.0 --port 5173`.
   - Плюсы: быстрая итерация, HMR (если разрешить).
   - Минусы: больше поверхность атаки; нужен строгий network policy.

**Рекомендация MVP:** только **static** для внешних пользователей; **dev** — флаг «доверенный пользователь» или только localhost через SSH-туннель.

### 5.2 Образ Docker (baseline)

- Базовый образ: `node:20-bookworm-slim`.
- Предустановленные CLI: `pnpm` или `npm`, опционально `playwright` только для CI-джобы без браузера в превью.
- Запуск от пользователя **non-root** (`uid 10001`).
- **read-only** root filesystem где возможно; **tmpfs** для `/tmp`; лимиты:
  - `--memory`, `--cpus`, `--pids-limit`, `--network` (см. ниже).

### 5.3 Сеть контейнеров

- По умолчанию **изолированная сеть** Docker `studio_isolated` без доступа к metadata cloud, внутренним IP VPS, Ollama.
- Разрешить исходящий **только**:

  - реестры npm (`registry.npmjs.org`), при необходимости зеркало;
  - фиксированный allowlist DNS (опционально через прокси).

- Запретить доступ к `169.254.169.254`, RFC1918 хосту (кроме явного proxy), `metadata.google.internal`.

### 5.4 Публичный URL превью

Варианты:

1. **Path-based**: `https://ollama.site-al.ru/preview/{token}/` → nginx `proxy_pass` на `127.0.0.1:{dynamic_port}` где слушает контейнер.
2. **Subdomain**: `https://{token}.preview.ollama.site-al.ru` — нужен wildcard TLS (Let’s Encrypt DNS-01).

Механика:

- При `POST /preview` orchestrator поднимает контейнер, биндует порт на **loopback** хоста `127.0.0.1:3xxxx`, региструет mapping `token → port` в Redis или локальном store.
- nginx upstream обновляется:

  - либо **lua/openresty** + shared dict (сложнее),
  - либо **отдельный маленький reverse proxy** (Caddy/traefik) с on-demand конфигом,
  - либо единый **HAProxy** с runtime map (как уже используется для SNI) — консистентно с вашей инфраструктурой.

**TTL:** превью живёт 15–60 минут неактивности; cron снимает контейнеры.

### 5.5 Ресурсы и стоимость

- Лимит параллельных превью на пользователя (например 1) и глобально (например 20).
- Очередь Fair queue (per user) чтобы один не забил весь хост.

## 6. Агент: инструменты (tool-calling)

Цель — чтобы агент не только «писал файлы», а **планировал**, **проверял** и **доводил** результат до уровня, близкого к продакшену: сборки, тесты, линт, базовый аудит безопасности и доступности, адаптивная вёрстка.

### 6.1 Минимальный набор инструментов для LLM (файлы и среда)

| Tool | Назначение |
|------|------------|
| `studio.read_file` | чтение файла из workspace (с лимитом размера) |
| `studio.write_file` | запись/создание |
| `studio.delete_path` | удаление файла/пустой папки |
| `studio.list_dir` | листинг |
| `studio.search` | **ripgrep** (`rg`) в контейнере по workspace — быстрый поиск по коду |
| `studio.run_command` | только **allowlist** (см. 6.2); расширенный список в 6.5 |
| `studio.request_preview` | триггер сборки static preview, возврат URL |
| `studio.read_build_log` | последние N строк stdout/stderr |

**Запрет:** произвольный `sh -c`, `curl`, `wget` без allowlist.

### 6.2 Реализация `run_command`

- Выполнение **только** внутри контейнера workspace (тот же образ, другой контейнер от одного snapshot).
- Timeout 3–10 минут (для **E2E** — отдельный лимит, напр. 15 мин), убийство по SIGKILL.
- Парсинг exit code; при ненулевом — сообщение агенту + лог; агент обязан **либо исправить**, либо явно зафиксировать блокер пользователю.

### 6.3 Планирование, согласование с пользователем и только потом реализация

**Правило:** до явного согласования пользователем агент **не** переходит к массовому изменению workspace и **не** запускает полный цикл сборки реализации. Детальный сценарий — **§1.6**.

| Tool / режим | Назначение |
|--------------|------------|
| `studio.plan_task` *(или фаза оркестратора)* | Разбить запрос на шаги: страницы, данные, API, риски; выход — план в чат + `StudioTask.status = awaiting_user_approval` |
| `studio.summarize_requirements` | Уточнить неоднозначности до фиксации плана |
| Шаблон **spec** в задаче / `SPEC.md` | Цели, не-цели, макеты, брейкпоинты, критерии готовности |
| **UI: согласование** | Кнопка «Согласовать и запустить» вызывает API (например `POST …/tasks/:taskId/approve-plan`) → `implementing` → разрешены `write_file` / `run_command` / финальный build |
| **UI: правки плана** | Отклонение или комментарий → возврат в `planning`, новая версия плана |

После согласования: **implementing** → **building** (lint/tsc/test/build по **§6.5**) → **ready_for_review** (preview URL + отчёт) → пользователь тестирует; следующая доработка — снова от **плана** (краткий план для микро-фикса допускается политикой, но должен быть явно показан пользователю).

### 6.4 Инструменты реализации, качества и «дороги к продакшену»

Команды ниже — кандидаты в **whitelist** для `studio.run_command` (точный список + версии фиксировать в образе runner).

**Сборка и типизация**

| Команда | Назначение |
|---------|------------|
| `npm ci` / `pnpm install --frozen-lockfile` | воспроизводимые зависимости |

**Статический анализ и стиль**

| Команда | Назначение |
|---------|------------|

| `npx prettier --check .` | единообразие форматирования (опционально `--write` только через отдельный доверенный шаг) |

**Тесты**

| Команда | Назначение |
|---------|------------|
| `npm run test` | **Vitest** (юнит/компоненты) |
| `npx vitest run --coverage` | покрытие; пороги — в конфиге проекта |
| `npm run test:e2e` | **Playwright** — смоук по критическим сценариям (в CI runner'а, не обязательно при каждом микро-изменении) |

**Зависимости и безопасность**

| Команда | Назначение |
|---------|------------|
| `npm audit --audit-level=moderate` | отчёт; блокирующий порог — политика |

| Tool | Назначение |
|------|------------|
| `studio.write_report` | краткий отчёт: что сделано, что проверено (lint/test/build), ссылки на preview, известные ограничения |

### 6.5 Definition of Done (по умолчанию для задач «фича/экран»)

Перед вызовом `request_preview` или пометкой задачи «готово» агент **должен** (если инструменты включены в шаблоне):

1. `npm run build` — успех.
2. `npm run lint` — успех (или явный waiver в отчёте с причиной).
3. `npm run test` — успех, если в шаблоне есть тесты; иначе — зафиксировать «тесты не в скоупе».
4. Проверить ключевые брейкпоинты **логически** (в промпте: перечислить, какие компоненты адаптивны; опционально скриншоты Playwright).
5. Для публичных форм — базовая a11y (labels, errors).

Пользователь может ослабить DoD для черновика (`draft`); для статуса `production_ready` — полный набор.

### 6.6 Политика зависимостей (популярные и проверенные пакеты)

- **Allowlist по категориям**: UI (Radix, Headless UI), данные (TanStack Query, Zod, Supabase JS), формы (**React Hook Form** + Zod), таблицы (**TanStack Table**) — список версий в `allowed-packages.json` в образе.
- Новые пакеты вне списка — **подтверждение в UI** или авто-разрешение только для scoped пакетов `@scope/*` после `npm audit`.
- Запрет известно-опасных паттернов: постинсталл-скрипты без ревью, зависимости с 0Maintainer без обоснования.

### 6.7 Промпт и модель

- Системный промпт: стек проекта, **1.4–1.5** (премиум-планка, Tailwind, адаптивность), ограничения по зависимостям.
- Vision: макеты пользователя — multimodal; агент сверяет отступы/иерархию с дизайном.
- Разделение: **планирование** (быстрая модель) → **код** (основная) → **фиксы по логам** (итерация) — pipeline в оркестраторе.

### 6.8 Связка с существующим чатом

- Опционально: кнопка «Открыть в чате» создаёт связь `project_id` в метаданных сессии чата.
- Или единый «контекстный объект» в БД: `ChatLink { type: 'studio', ref_id }`.

## 7. Безопасность (все тонкости)

### 7.1 SSRF и preview

- Превью не должно дергать внутренние сервисы по IP пользователя (не применимо к static dist, но применимо если в будущем embed внешние URL в iframe).

### 7.2 CSP и iframe

- Превью на **отдельном origin** (`preview.*`) облегчает изоляцию.
- Заголовки: `Content-Security-Policy` по умолчанию строгие; для пользовательского HTML — sandboxed **subframe** с `sandbox` атрибутами на родителе + изолированный origin.

### 7.3 Supply chain

- `npm audit` в CI-пайплайне агента (информировать пользователя).
- Опционально: запрет install новых пакетов без подтверждения UI.

### 7.4 Секреты

- Никакие `.env` с реальными ключами в ревизиях по умолчанию; шаблон `.env.example`.
- Если нужны ключи пользователя — отдельное зашифрованное хранилище (KMS или libsodium с ключом сервера).

### 7.5 Утечки между пользователями

- Обязательная проверка `project.user_id === session.user_id` на каждом запросе.
- Runner подписывает JWT на один `project_id` + `revision_id` + `exp`.

## 11. Поэтапный план внедрения

### Фаза 0 — уже есть

SPA маршрут `/lovable`, статический каркас UI.

### Фаза 1 — «пассивный проект»

- CRUD проектов, шаблон `vite-react-ts` из архива.
- Загрузка/чтение файлов через API (**реализовано:** `GET/PUT …/files`, `GET …/files/content`).
- **§4.2 ревизии и workspace** (**реализовано:** `GET/POST …/revisions`, tar.gz + meta; `PATCH …/workspace` с проверкой `base_revision_id` / head и только при `plan.status === 'approved'`; UI — блок «Ревизии», снимок, HEAD).
- Ручное редактирование одного файла в UI (опционально) — **в `/lovable` список файлов + редактор**.

### Фаза 2 — runner static preview

- Docker образ, сборка `npm ci && npm run build`.
- Выдача URL через nginx map.
- TTL и лимиты.

### Фаза 3 — агент

- Интеграция tool-calling с существующим LLM gateway (**частично:** стрим + нативный tool API Ollama — следующий шаг; **есть:** оркестрация `implement` → JSON-операции → тот же `PATCH workspace`, события SSE `tool_start` / `tool_end` / `revision`).
- SSE стрим в UI (**реализовано:** `POST …/agent/run`, `GET …/agent/stream/:runId`, события `delta`, `plan_ready`, `error`, `done`; LLM через существующий Ollama `/api/generate` stream).
- Задачи `StudioTask` + история (**реализовано:** `GET/POST …/tasks`, `POST …/tasks/:taskId/approve-plan`; UI блок «Задачи §4.3»).
- Включить **фазу планирования** до массовых правок файлов (**реализовано:** режим `plan` у `agent/run`; **`implement`** после апрувов: пояснение в потоке + второй вызов модели с JSON → `studioApplyWorkspacePatchTxn`, до `STUDIO_AGENT_IMPLEMENT_MAX_OPS` операций).

### Фаза 4 — качество и «премиум-пайплайн»

- В runner: **lint** (ESLint), **tsc**, **vitest**, при необходимости **Playwright** смоук и **axe**.
- Отчёт агента в UI: build/lint/test зелёные или приложен лог.
- **npm audit** в отчёте; порог блокировки preview — по политике.
- Diff UI, откат ревизии.
- Шаблоны новых проектов: **Tailwind + TS + TanStack Query** и миграция metadata студии к **Postgres/Supabase** (параллельно с JSON при необходимости).

### Фаза 5 — масштаб

- Вынос worker, очередь Redis, HA preview proxy.
- Расширение allowlist пакетов и отдельный **Lighthouse CI** / бюджеты на staging preview.

## 12. Изменения nginx (черновик)

Дополнительно к существующему `location /api/`:

```nginx
# Пример: превью по пути (токен проверяется в studio-api перед выдачей upstream)
location ~ ^/preview/(?<ptoken>[a-zA-Z0-9_-]+)(?<puri>/.*)$ {
    proxy_pass http://127.0.0.1:3020/internal/preview/$ptoken$puri;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_read_timeout 300s;
}
```

Точная схема зависит от выбора **path** vs **subdomain** и от того, где живёт routing (nginx vs sidecar).

## 15. Решения «зафиксировать на ревью»

Перед кодированием утвердить:

1. Path-based vs subdomain для preview.
2. Git внутри сервера vs только tar-ревизии.
3. Один процесс Node vs отдельный `studio-api`.
4. Политика зависимостей (allow all public npm vs **allowlist по категориям**, см. 6.6).
5. **СУБД для Studio:** PostgreSQL self-hosted vs **Supabase** vs этапно (SQLite/JSON → Postgres).
6. Обязательный **Definition of Done** для задач: полный набор (build+lint+test) vs облегчённый для черновиков.
7. Включать ли **Playwright + axe** в каждый preview-run или только в «release» задачах.

## 16. Рекомендации к следующим итерациям плана

Дополнения к основному тексту — по мере созревания продукта вносить в реализацию и при необходимости разбивать на отдельные задачи в бэклоге.

### 16.1 Продукт и UX

- **Отмена / стоп run** — явное действие пользователя: прервать долгий прогон агента или сборку без оставления «висящих» контейнеров; статус задачи `cancelled` с причиной в логе.
- **Сравнение с предыдущей ревизией** — помимо diff в чате: список изменённых путей, опционально два preview (до/после) или одна страница «что изменилось» для приёмки.
- **Каталог шаблонов проектов** — короткие пресеты с описанием (лендинг, личный кабинет, форма + БД): ускоряют план агента и делают объём работ предсказуемым для пользователя.
- **Чек-лист UX студии** (для приёмки UI самой `/lovable`): адаптивный layout панелей, пустые состояния, ошибки сети и таймаутов, доступность базовых кнопок (согласование плана, отмена).

### 16.2 Архитектура и надёжность

- **Идемпотентность** — повторный `approve-plan` или дубль `agent/run` не создаёт второй параллельный прогон без явного намерения; использовать ключ идемпотентности или уникальный индекс «одна активная run на task».
- **SLA в продуктовых терминах** — зафиксировать ожидания (например время до первого preview при пустой очереди); при сбое — retry, понятное сообщение пользователю, запись для саппорта.
- **Версионирование системного промпта и политик агента** — хранить версию/хеш в `AgentRun` или конфиге, чтобы сравнивать качество между релизами бэкенда.

### 16.3 Безопасность и соответствие

- **ToS и пользовательский контент** — превью и сгенерированные приложения: правила допустимого контента, канал abuse, процедура блокировки; учёт рисков вредоносного или вводящего в заблуждение кода в пользовательских проектах.
- **Rate limits** — не только на общий API: лимиты на число **согласованных** планов / активных run в сутки на пользователя, защита от злоупотребления LLM и runner.

### 16.4 Наблюдаемость и метрики

- **Сквозная корреляция** — `task_id`, `run_id`, `project_id` в логах фронта (события UI), gateway и runner; один запрос в поиске по дежурной смене.
- **Продуктовые метрики** — доля задач с планом без возврата на доработку, медианное время до `ready_for_review`, доля failed build, причины отказов (агрегировано).

### 16.5 Документация и согласованность в команде

- **Матрица принятых решений** — таблица: тема (preview path, СУБД, allowlist), вариант, дата/версия «принято», ссылка на PR/issue; дополнять по мере закрытия пунктов §15.
- **Глоссарий** — краткие определения: *revision*, *preview*, *run*, *plan*, *approve*; уменьшает разночтения между фронтом, бэком и продуктом.
